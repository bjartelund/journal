# Journal App (Nitrogen)

This is an Erlang web application built on the [Nitrogen Web Framework](https://nitrogenproject.com).
It provides a simple journal entry page (see `src/index.erl`) that writes timestamped text files under
`priv/journal/` (these are ignored by git so deployments start clean).

---
## Prerequisites

You need the following installed on any machine that clones this repo:

* Erlang/OTP 27 (other recent OTP versions may work, but 27 is what this was built/tested with)
* git (used both for cloning and for the version script to derive build metadata)
* curl (only if `rebar3.mk` needs to fetch `rebar3` the first time)
* Optional: openssl (for `make cookie` to generate a node cookie)

No global Nitrogen install is required; dependencies are pulled via `rebar3` using the committed `rebar.lock`.

---
## Quick Start

Clone and run a live dev shell with hot code reloading:

```
git clone <your-repo-url> journal
cd journal
make run_dev
```

Then visit: http://localhost:8000/

To build and run a managed release (daemonized):

```
make daemon
make attach   # Attach an Erlang shell to the running release
```

Stop the release (from another shell):

```
_build/default/rel/journal/bin/journal stop
```

## Development vs Release Modes

| Mode | Command | Characteristics |
|------|---------|-----------------|
| Dev Shell | `make run_dev` | Assembles config, starts `rebar3 shell`, launches Sync for live reload |
| Release (daemon) | `make daemon` | Builds relx release, version-bumps, runs in background |
| Release (interactive) | `make run_release` | Runs release foreground with an Erlang shell |
| Attach | `make attach` | Connect to already started daemon |

Live reload is powered by the `sync` application automatically started in `run_dev` (see Makefile target).

## Versioning Workflow

The build uses `make_version_file.escript` to manage version files:

* `version.current` – currently active version
* `version.previous` – previous version
* `version.history` – timestamped history of transitions
* `version.lock` – temporary lock while a release is assembling

`make daemon` (or any `make release`) sequence:
1. `./make_version_file.escript go` (generates or advances version + creates `version.lock`)
2. Assembles combined config (`etc/assemble_config.escript` → `etc/autogenerated.config.all`)
3. Builds relx release
4. `make finish_version` (calls `./make_version_file.escript finish` to append to history + remove lock)

If a build fails mid-way and leaves a `version.lock`, recover with:
```
make revert_version
```

## Configuration

* Primary runtime configuration lives in `etc/*.config`.
* `etc/simple_bridge.config` defines the HTTP backend (`cowboy`), port (default 8000), and static paths.
* `etc/app.config` includes Nitrogen application settings and SASL logging setup.
* The Makefile’s `run_dev` target now assembles and loads `etc/autogenerated.config.all` to ensure the backend is defined during development.

Change port or address by editing `{port, 8000}` / `{address, "0.0.0.0"}` in `etc/simple_bridge.config` and re-running `make run_dev` or `make daemon`.

## Adding Features

Typical flow for adding a page:
1. Create `src/my_page.erl` exporting `main/0` (and optionally `title/0`, `body/0`).
2. Hit `http://localhost:8000/my_page` (dynamic route resolution).
3. Use Nitrogen elements (`#h1{}`, `#panel{}`, etc.) defined in `nitrogen_core` headers (`wf.hrl`).

See existing `src/index.erl` for a minimal example.

## Deployment Notes

For production:
1. Build: `make daemon` (or `PROFILE=prod make daemon` if you later add profiles).
2. Use `nginx` (see `nginx/nitrogen.nginx` template) as reverse proxy / TLS terminator.
3. Persist logs and journal data if desired (remove ignores in `.gitignore` or mount a volume).

To stop the release:
```
_build/default/rel/journal/bin/journal stop
```

To check status:
```
_build/default/rel/journal/bin/journal ping
```

## TROUBLESHOOTING

| Symptom | Cause | Fix |
|---------|-------|-----|
| `simple_bridge ... 'no backend defined'` | Dev shell started without assembled config | Use `make run_dev` (already fixed) |
| `Cannot proceed. Previous version attempt failed.` | Stale `version.lock` | `make revert_version` |
| Port already in use | Another process on 8000 | Change port in `etc/simple_bridge.config` |
| Crash on missing module | Edited module not recompiled | In dev shell ensure `sync:go().` displayed |

## Original Minimal Notes (for reference)

### Original Building Section

You can rebuild the project by typing:

   make

If this fails, it likely means that its a slim release and Erlang is not
installed on this machine. Please [install
Erlang](https://www.erlang-solutions.com/resources/download.html).

### Original Running Section

You can start the application with:

  make daemon

which will start it in daemon mode.  You can attach to this started daemon
with:

  make attach

If you'd rather have an Erlang console right away, rather than starting a
daemon, type:

  make run_release

You can hot-upgrade a running release with:

  make upgrade_running

If a release fails to build or upgrade, you can revert the current version with

  make revert_version

### Original Config Section

The rebar.config file controls dependencies and build parameters using the
popular [rebar3](http://rebar3.org) tool.

The `plugins.config` file controls how Nitrogen plugins are handled.

The files in the `etc` directory control application configuration, including
ports, webserver set up, and the like.  Please see the files there for
configuration options, or view the Nitrogen documentation.

### Original Code Section

Source code can be found in `src`

Template code is in `priv/templates`

Static resources (images, js, etc) go in `priv/static`

### Original Live code reloading Section

By default Nitrogen comes with an application called
[sync](http://github.com/rustyio/sync) which provides automatic code reloading
when the related .erl file or .ebin file is updated.  To use this, run the
following on the Erlang shell:

  sync:go().
